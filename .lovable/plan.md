## Diagnosis

Inspected the latest "Mux testing" post in the DB. Its `media` JSON is:

```json
{
  "id": "...", "type": "video", "order": 0,
  "url": ".../<uuid>_poster.jpg",
  "thumbnail_url": ".../<uuid>_poster.jpg",
  "alt": "IMG_2408", "caption": "", "is_deleted": false
}
```

So:
1. `provider:'mux'` — **MISSING**
2. `mux_upload_id` — **MISSING**
3. `mux_status:'preparing'` — **MISSING**
4. `url` = poster URL ✓
5. `thumbnail_url` = poster URL ✓
6. The `.MOV` **did** go through the Mux branch (url and thumbnail are the poster jpg generated by `uploadVideoViaMux`, not the original video). The Mux fields are simply being stripped before the row is written.

### Where the fields are stripped

`src/components/feed/EnhancedCreatePostForm.tsx` lines 515–524:

```ts
const mediaToSave = media.map(item => ({
  id: item.id || uuidv4(),
  url: item.url,
  type: item.type,
  caption: item.caption || '',
  alt: item.alt || '',
  order: item.order,
  is_deleted: false,
  thumbnail_url: item.thumbnail_url || item.url,
}));
```

This whitelist predates Phase 2A/2B and silently drops `provider`, `mux_upload_id`, `mux_asset_id`, `mux_playback_id`, `mux_status`, `mux_error`, plus `width`/`height`/`duration`/`orientation`.

### Why the feed shows "This video format isn't supported on your device"

With `provider` missing on the persisted item:
- `isMuxPreparing(item)` returns `false` (Phase 2A guard correctly checks `provider === 'mux'`)
- The feed renderer falls back to a native `<video>` element with `src = url = poster.jpg`
- Browser tries to decode a JPEG as a video → "format not supported"

The Phase 2A guards in `FeedVideo`, `LightboxPreview`, `MediaGallery`, `TwitterStyleMediaPreview`, etc. are all correct. The composer preview looked OK-ish because in-memory `media` state still carried `provider:'mux'`, so `FeedVideo` used `MuxPreparingPoster` there. (Any "looks bad" perception is just the portrait poster letterboxed into the composer tile — not a bug.)

### Mux upload itself

The `.MOV` was routed through `uploadVideoViaMux` (otherwise `url` wouldn't be a `_poster.jpg`). The PUT and `mux_upload_success` analytics presumably fired client-side. Whether a `mux_uploads` row exists for that `upload_id` is unknowable from the persisted post because `mux_upload_id` was discarded — we'll be able to verify this for the next upload once the fix lands.

## Fix (Phase 2B completion — not Phase 3)

This is a Phase 2B gap, not a Phase 2A renderer bug and not something Phase 3/4 was supposed to address. Single, narrow change:

**Edit `src/components/feed/EnhancedCreatePostForm.tsx`** — extend the `mediaToSave` whitelist to pass through the Mux fields (and the geometry fields that Phase 2A renderers rely on) when present:

```ts
const mediaToSave = media.map(item => ({
  id: item.id || uuidv4(),
  url: item.url,
  type: item.type,
  caption: item.caption || '',
  alt: item.alt || '',
  order: item.order,
  is_deleted: false,
  thumbnail_url: item.thumbnail_url || item.url,
  ...(item.width ? { width: item.width } : {}),
  ...(item.height ? { height: item.height } : {}),
  ...(item.duration ? { duration: item.duration } : {}),
  ...(item.orientation ? { orientation: item.orientation } : {}),
  ...(item.provider ? { provider: item.provider } : {}),
  ...(item.mux_upload_id ? { mux_upload_id: item.mux_upload_id } : {}),
  ...(item.mux_asset_id ? { mux_asset_id: item.mux_asset_id } : {}),
  ...(item.mux_playback_id ? { mux_playback_id: item.mux_playback_id } : {}),
  ...(item.mux_status ? { mux_status: item.mux_status } : {}),
  ...(item.mux_error ? { mux_error: item.mux_error } : {}),
}));
```

Conditional spreads keep legacy Supabase items byte-identical (no `provider:null` noise in old JSON).

**Nothing else changes.** No schema migration (column is `jsonb`), no renderer changes, no service changes, no analytics changes. Flag stays `true` only for verification, then back to `false` at Gate 4 as previously agreed.
